# OCR Worker Design

## Background

Evidence documents (medical records, invoices, EOBs) arrive with a `storage_uri` pointing to a blob store. Before the FWA scoring pipeline can use their text content, each document must be OCR-processed and the structured output registered against `evidence_ocr_outputs`. This design adds a standalone `apps/ocr-worker` binary backed by a `crates/fwa-ocr` shared crate.

## Goals

- Consume OCR tasks from a message queue and write results back via the existing `POST /api/v1/ops/evidence/documents/{id}/ocr-outputs` endpoint.
- Keep queue backend and OCR provider swappable via traits; selection is config-driven.
- Run in CI and demo environments without any external dependencies (stub + noop implementations).

## Non-Goals

- No retry counter in the worker itself — retry responsibility belongs to the queue.
- No PII extraction or redaction pipeline.
- No embedding job triggering (separate concern, separate worker command if needed later).

## Directory Layout

```
crates/fwa-ocr/
  Cargo.toml
  src/
    lib.rs           re-exports, OcrTask, OcrResult
    queue.rs         MessageQueue trait + InProcessStub
    queue_redis.rs   RedisQueue
    queue_sqs.rs     SqsQueue
    provider.rs      OcrProvider trait + HttpOcrProvider + NoopOcrProvider
    processor.rs     OcrProcessingLoop
    writeback.rs     WritebackClient (reqwest → API server)

apps/ocr-worker/
  Cargo.toml
  src/
    main.rs          env config + driver selection + loop start
```

`fwa-ocr` has no dependency on `apps/api-server`. It calls the API server over HTTP via `reqwest`.

## Trait Interfaces

### MessageQueue

```rust
#[async_trait]
pub trait MessageQueue: Send + Sync {
    async fn receive(&self) -> anyhow::Result<Option<OcrTask>>;
    async fn ack(&self, receipt: &str) -> anyhow::Result<()>;
    async fn nack(&self, receipt: &str) -> anyhow::Result<()>;
}

pub struct OcrTask {
    pub receipt: String,
    pub document_id: String,
    pub storage_uri: String,
    pub customer_scope_id: String,
    pub claim_id: Option<String>,
}
```

**InProcessStub**: `Arc<Mutex<VecDeque<OcrTask>>>`. `receive()` pops front; `ack()` and `nack()` record calls for test assertions.

**RedisQueue**: `BRPOP` on a configurable list key. `nack()` re-pushes to a dead-letter list.

**SqsQueue**: `ReceiveMessage` with `WaitTimeSeconds=20`. `ack()` calls `DeleteMessage`; `nack()` lets visibility timeout expire.

### OcrProvider

```rust
#[async_trait]
pub trait OcrProvider: Send + Sync {
    async fn process(
        &self,
        document_bytes: &[u8],
        mime_type: &str,
    ) -> anyhow::Result<OcrResult>;
}

pub struct OcrResult {
    pub engine: String,
    pub engine_version: String,
    pub output_text: String,
    pub confidence_score: Option<f64>,
    pub quality_status: String,  // "accepted" | "low_confidence" | "failed"
}
```

**NoopOcrProvider**: Returns fixed `output_text = "[noop ocr output]"`, `quality_status = "accepted"`. Used in CI and demo.

**HttpOcrProvider**: POSTs `multipart/form-data` to `OCR_ENDPOINT`. Expects JSON response `{ text, confidence, engine_version }`. Plugs into any REST OCR service (AWS Textract adapter, Azure Form Recognizer adapter, or custom).

## Processing Loop

`OcrProcessingLoop` holds `queue`, `provider`, `writeback`, and `poll_interval`.

```
loop:
  match queue.receive():
    None  → sleep(poll_interval); continue
    Some(task):
      bytes   = http_get(task.storage_uri)
      result  = provider.process(bytes, infer_mime(task.storage_uri))
      out_uri = encode_inline_uri(task.document_id, &result.output_text)
      writeback.post_ocr_output(&task, &result, &out_uri)
        ok  → queue.ack(task.receipt)
        err → queue.nack(task.receipt); log error; continue
```

Panics are not propagated — each task error is logged and the loop continues. The binary exits only on SIGTERM/Ctrl-C (tokio `ctrl_c()` signal).

**`output_uri` encoding**: `ocr://inline/{document_id}/{ocr_output_id}` where `ocr_output_id` is a UUIDv4 generated by the worker. The `output_text` is stored base64-encoded in the `output_uri` field of `EvidenceOcrOutputRecord`. When the client environment provides object storage, `upload()` writes the text to S3/Blob and returns the real URI; the loop is unchanged.

## Writeback

`WritebackClient` wraps a `reqwest::Client`. One method:

```rust
pub async fn post_ocr_output(
    &self,
    task: &OcrTask,
    result: &OcrResult,
    output_uri: &str,
) -> anyhow::Result<()>
```

POSTs to `{API_URL}/api/v1/ops/evidence/documents/{document_id}/ocr-outputs` with `x-api-key: {API_KEY}`. Maps directly onto the existing `CreateEvidenceOcrOutputRequest` shape.

## Configuration (apps/ocr-worker)

| Variable | Values | Default |
|---|---|---|
| `QUEUE_DRIVER` | `stub` \| `redis` \| `sqs` | `stub` |
| `QUEUE_URL` | Redis URL or SQS queue URL | — |
| `OCR_PROVIDER` | `noop` \| `http` | `noop` |
| `OCR_ENDPOINT` | URL of HTTP OCR service | — (required when `http`) |
| `API_URL` | API server base URL | required |
| `API_KEY` | API server key | required |
| `POLL_INTERVAL_MS` | milliseconds | `2000` |

`QUEUE_DRIVER=stub` + `OCR_PROVIDER=noop` is valid for smoke-testing the binary without any external services.

## Testing Strategy

- **Unit tests in `fwa-ocr`**: `InProcessStub` + `NoopOcrProvider` drive `OcrProcessingLoop`. Verify: task is acked on success, nacked on writeback error, loop continues after error, ack/nack call counts match.
- **IO adapters** (`RedisQueue`, `SqsQueue`, `HttpOcrProvider`): no unit tests. Verified by integration tests in client environments.
- **CI cadence**: `cargo check --locked -p fwa-ocr`, then `cargo test --locked -p fwa-ocr`. Follows cadence B from CLAUDE.md.
